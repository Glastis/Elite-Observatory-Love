2. **NO if/elseif chains** - Use lookup tables

## Project Description

**Elite Observatory (LÖVE port)** — a Lua/LÖVE 11.x port of
[Elite Observatory Core](https://observatory.xjph.net/) (originally a C# /
Windows Forms app). Cross-platform (Windows, Linux, macOS) live monitor of
the Elite Dangerous journal folder, with a Lua plugin system, an
immediate-mode UI, native popup notifications, audio playback and
TSV/CSV export.

Key entry points:
- `main.lua` — LÖVE entry point (`love.load`, `love.update`, `love.draw`).
- `conf.lua` — LÖVE config (window, modules).
- `observatory/*.lua` — core modules (paths, settings, journal_reader,
  log_monitor, audio_handler, notifications, export_handler,
  plugin_manager, core).
- `observatory/ui/*.lua` — theme, immediate-mode widgets, core form.
- `plugins/<id>/init.lua` — Lua plugins discovered at startup
  (`plugin_manager.load_all`).
- `tests/run.lua` — pure-Lua unit tests (no LÖVE runtime required).
- `lib/json.lua` — rxi/json.lua (MIT).

## Tech Stack
- Framework: **LÖVE 11.x** (liblove)
- Language: **Lua 5.1 / LuaJIT** (LÖVE's embedded runtime). Tests run
  under any standalone `lua` / `luajit` interpreter.
- Modules: native `require()` system — modules return a table with the
  public API.
- Persistence: settings stored as JSON in the LÖVE save dir
  (`observatory.config`).
- Assets: images, sounds, fonts via LÖVE APIs.
- No build step. Run with `love .` (or `love . --smoke` for the
  non-graphical CI check).

## Running and testing

```sh
# Launch the app
love .

# Headless smoke test (loads plugins + journals, prints a summary)
love . --smoke
love . --smoke --journal /path/to/Elite/Dangerous

# Pure-Lua unit tests (no LÖVE needed)
lua tests/run.lua
```

Press `F5` to reload plugins, `Esc` to quit.

## Code Conventions

### Code structure
- Avoid ad hoc (case-by-case) processing. Prefer generic approaches.
- Limit function size to **30-40 lines**.
- Split into subfunctions to improve readability and clarify steps.
- Reduce indentation level:
    - Handle errors early with `return`
    - Isolate code into subfunctions
- Keep files concise. If a file gets too long, rethink the architecture.
- Avoid code duplication. Extract common parts into parameterized functions.
- No magic numbers, magic strings, or any hardcoded logic values.
- Separate logic and definitions (constants in their own place, no scattered definitions in logic).

#### Examples

Readability
```lua
-- bad: 120 lines, all in one
function process_order(order)
    validate(order)
    -- ... tax calculations, stock, payment, email, logs ...
end

-- better: reader can follow the flow
function process_order(order)
    validate(order)
    local priced = price_order(order)
    reserve_stock(priced)
    charge_payment(priced)
    notify_customer(priced)
end
```

Reduce indentation
```lua
-- Deep indentation
function get_user_name(id)
    local user = db.find(id)
    if user then
        if user.email then
            local history = get_user_history(user.email)
            local orders = extract_orders(history)
            return user.name
        end
    end
    return "unknown"
end

-- Early return
function get_user_name(id)
    local user = db.find(id)

    if not user or not user.name then
        return "unknown"
    end

    local history = get_user_history(user.email)
    local orders = extract_orders(history)
    return user.name
end
```

### Flow management

**NEVER use `if/elseif` chains for known values**

Instead, use:
- Lookup tables
- Loops with early returns
- Callback/handler patterns
- Injected strategy functions

#### Examples

Multiple similar calls
```lua
-- bad
if has_role(user, "admin") then grant("all") end
if has_role(user, "editor") then grant("write") end
if has_role(user, "viewer") then grant("read") end

-- bad
if user.role == "admin" then
    grant("all")
elseif user.role == "editor" then
    grant("write")
elseif user.role == "viewer" then
    grant("read")
end

-- good: logic and config split
local grants_by_role = {
    admin = "all",
    editor = "write",
    viewer = "read"
}

for _, role in ipairs(user.roles) do
    local permission = grants_by_role[role]
    if permission then
        grant(permission)
    end
end
```

Multiple similar conditions
```lua
-- bad
function handle_role(user)
    if user.role == "admin" then
        return "Full access"
    elseif user.role == "editor" then
        return "Can edit content"
    elseif user.role == "viewer" then
        return "Can only read"
    elseif user.role == "guest" then
        return "Limited access"
    else
        return "Unknown role"
    end
end

-- good
local role_messages = {
    admin = "Full access",
    editor = "Can edit content",
    viewer = "Can only read",
    guest = "Limited access"
}

function handle_role(user)
    return role_messages[user.role] or "Unknown role"
end
```

Function dispatcher
```lua
-- good: handler table
local input_handlers = {
    escape = function() love.event.quit() end,
    space = function() player:jump() end,
    left = function() player:move_left() end,
    right = function() player:move_right() end
}

function love.keypressed(key)
    local handler = input_handlers[key]
    if handler then
        handler()
    end
end
```

Journal events follow the same pattern — a plugin should dispatch
`entry.event` strings via a table, never via an `if/elseif` chain.

### Naming

- A variable name must indicate **its role** or **its content**, not its type.
    - Correct examples: `count`, `amount`, `player_position`.
- Booleans must be explicit and start with `is_`, `should_`, `must_`, `has_` or `can_`.
    - Examples: `is_active`, `should_retry`, `has_ammo`, `can_jump`.
- Use **snake_case** for everything (variables, functions, files).
- Adapt clarity to context:
    - limited scope → short name
    - wide scope → explicit name

#### Examples

Variables
```lua
-- bad: not literal
local str1 = "Bob"
local arr_ord = {'abc-sd3', 'abc-df2'}

-- good
local customer_name = "Bob"
local orders = {'abc-sd3', 'abc-df2'}

-- good
local customer = {
    name = "Bob",
    orders = {'abc-sd3', 'abc-df2'}
}
```

Booleans
```lua
-- bad
local active = true

-- good
local is_active = true
local should_retry = false
local has_jumped = true
local can_move = false
```

Functions naming
```lua
-- bad: not explicit
function debug_user(user)
    print(user.name, user.cart)
end

-- good
function print_user_info(user)
    print(user.name, user.cart)
end

-- bad: not clear
function process_orders(user)
    -- business logic
end

-- good
function validate_user_orders(user)
    -- business logic
end
```

Range
```lua
-- Small range, small code block
for _, item_id in ipairs(item_ids) do end

-- Small range, small code block, i is ok
local i = 0
while i < max_iterations do
    i = i + 1
end

-- Large range, requires more info
local last_registered_user_id = 'abc-ds3-fsd'
```

Count
```lua
local retry_count = 0
local item_amount = 5

-- depending on context:
local attempts
local failed_attempts
```

### Comments

**ABSOLUTELY NO COMMENTS IN CODE - ZERO TOLERANCE**

- NO inline comments
- NO block comments
- NO explanatory comments (make self-explanatory functions and subfunctions instead)
- NO docstrings
- Code MUST be self-documenting through clear naming and structure
- If you think a comment is needed → REFACTOR into subfunctions with explicit names

### LÖVE structure
```lua
-- constants.lua
return {
    WINDOW_WIDTH = 800,
    WINDOW_HEIGHT = 600,
    PLAYER_SPEED = 200,
    GRAVITY = 500
}

-- main.lua
local constants = require("constants")
local player = require("entities.player")

function love.load()
    player.init(constants.WINDOW_WIDTH / 2, 100)
end

function love.update(dt)
    player.update(dt)
end

function love.draw()
    player.draw()
end

function love.keypressed(key)
    player.handle_key(key)
end
```

### Design principles

- Be as generic as possible. Each logic part must be as agnostic as possible of other parts.
- Follow **DRY** (Don't Repeat Yourself) principle every time, no exception.
- Follow **KISS** (Keep It Simple, Stupid) principles as often as you can.
- Keep files and functions small, separate constants and interfaces in their dedicated files.
- Use Lua modules with `return` to expose the public API.
- Plugins must only depend on the documented core API in
  `observatory/core.lua` — never reach into other plugins or LÖVE
  internals directly.

### Module organization
```lua
-- entities/player.lua
local constants = require("constants")

local player = {}
local position = { x = 0, y = 0 }
local velocity = { x = 0, y = 0 }

function player.init(x, y)
    position.x = x
    position.y = y
end

function player.update(dt)
    apply_gravity(dt)
    apply_movement(dt)
    clamp_position()
end

function player.draw()
    love.graphics.circle("fill", position.x, position.y, 20)
end

function apply_gravity(dt)
    velocity.y = velocity.y + constants.GRAVITY * dt
end

function apply_movement(dt)
    position.x = position.x + velocity.x * dt
    position.y = position.y + velocity.y * dt
end

function clamp_position()
    position.x = math.max(0, math.min(constants.WINDOW_WIDTH, position.x))
    position.y = math.max(0, math.min(constants.WINDOW_HEIGHT, position.y))
end

return player
```

## Plugin contract

Plugins live under `plugins/<id>/init.lua` and must `return` a table
exposing:

- `id`, `name`, `short_name`, `version` — string identifiers
- `grid = { columns = {...}, rows = {} }` — UI grid descriptor
- `default_settings` — table merged into persistent settings
- `Plugin:load(core)` — receive the core API (see `observatory/core.lua`)
- `Plugin:journal_event(entry)` — called on every parsed journal line
- `Plugin:status_change(status)` — called when `Status.json` changes
- `Plugin:on_notification(args)` — called on emitted notifications

See `plugins/example/init.lua` for a working reference.
